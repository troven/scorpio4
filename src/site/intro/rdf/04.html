Inferring Terms

Scorpio4 natively understands the RDFS vocabulary. In practice, that means Scorpio4 can apply common sense and infer:

        :sally      a       eg:Person.
        :harry      a       eg:Person.

It doesn't quite end there. The RDFS vocabulary itself defines inferences. We can expect to infer some basic facts such as:

    eg:Person
                    a       rdfs:Class;
                    a       rdfs:Resource.

    :name           a       rdf:Property.
    :DoB            a       rdf:Property.
    :knows          a       rdf:Property.

Term Hierarchies

RDFS has two (2) other terms that are similarly powerful rdfs:subClassOf and rdfs:subPropertyOf

We might want to state that all instances of a eg:Male or a eg:Female are also a type of eg:Person. It's a reasonably safe assumption.

        eg:Male     rdfs:subClassOf     eg:Person.
        eg:Female   rdfs:subClassOf     eg:Person.

        :anne       a                   eg:Female.
        :dick       a                   eg:Male.

The RDFS inferencing rules would apply "common sense" and automatically assert some : 

        :anne      a       eg:Person.
        :dick      a       eg:Person.

Other hierarchies and complex data structures can be built up from these basic terms.

Nonesense

Before we get carried away, please remember that neither RDF nor Scorpio4 are smarter than we are.

It's quite easy to make some logical gaffes that the RDFS inferencing rules will take as facts, and therefore drawn the wrong conclusions.

In more advanced exercises, we can define Sanity Checks that can identify and flag these logic errors.

Open and Closed World

A SQL schema frequently asserts constraints that limit what can, and must, be described. This works very well in a closed system, such as a legacy business application.

RDFS, however, works in a more uncertain world. It assumes that not all information is present. The RDFS schema instead specifies inferences, logical statements that can be assumed when certain statements are present.
