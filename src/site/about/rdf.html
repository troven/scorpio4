Scorpio4 RDF

Scorpio4 let you works in a near-english language, that grows smarter the most it's used.

It's a progeny of Sir Tim Berners Lee, it's call RDF. And in a certain light it is rather elegant.

RDF is a data modelling language that uses URIs to describe everything. A URI is a globally agreed unique identifier. Any web site URL is also a URI.

We can make english-like statements using 3 URIs using the Turtle syntax:

        @PREFIX eg:     <http://example.org/>.              # a prefix makes URIs prettier

        eg:Sally         eg:drinks          eg:Coffee.      # the full stop is required.

If we said the same thing without the PREFIX, we would see:

        <http://example.org/Sally>
        <http://example.org/drinks>
        <http://example.org/Coffee>.

Hard Facts

A "Hard Fact" is comprised of 3 distinct URIs. In RDF, they are called the subject, predicate and object.

The predicate is the powerhouse. It tells us how the subject and object are related. It's quite helpful to use it like a verb or adjective.

        eg:Sally         eg:knows           eg:Bob.
        eg:Bob           eg:drinks          eg:Tea.

Unlike with literals, you can make further statements about each URI independently. This allows us to create a 3 dimension data model.

Literal Facts

Literals can represent strings, booleans, numbers, dates, times. Any simple xsd: data type.

 eg:Sally    eg:name         "Sally Smith"^^xsd:string.

Where we have declared that resource eg:Sally has a literal "Sally Smith" as her eg:name.

Our fact is semantically weaker, since we can't state anything facts about the literal itself.

Fact Space

Often we want to define triples for a given context, a problem domain. The same facts may not hold true under different scenarios, or circumstances.

We call this a "Fact Space", and it's referenced by a URI, of course. Painlessly, our triples are now quads. We are working with a 4 dimension data model.

Inferencing

If you think about relationships, they tell you a lot about the two parties. In other words, we can make inferences about the subjects and objects by referencing the predicate.

If we use the rdfs vocabulary to describe predicates:.

        eg:drinks       rdfs:domain         eg:Drinker.
        eg:drinks       rdfs:range          eg:Beverage.

Scorpio4 will logically infer the following statements:

        eg:Tea           a                  eg:Beverage.
        eg:Coffee        a                  eg:Beverage.
        eg:Bob           a                  eg:Drinker.
        eg:Sally         a                  eg:Drinker.

Linked Data

Each atomic element of the statement is actually a URI - a globally unique identifier just like a web page. In fact some web pages already support embedded RDF.

http://example.org/whatis/Black_Coffee

Since each URI is globally addressable, it's possible to request more RDF that further expands the definition of :BlackCofee - perhaps that is suppliers, recipes or health related

From the RDF that the Black_Coffee site provides about it's beverage, we find more statements. We can continue like this until all the world's relevant linked data is consumed.

It's the links that matter. Those relationships provide meaning - hence the semantic web. Since all data can theoretically be called upon to explain itself, it makes for a very powerful knowledge management framework.

To Schema or Not Schema

A schema in SQL is a set of constraints that are imposed on the structure of the data. With RDF, a schema is a logical set of rules that infer further facts.

Furthermore, the data and meta-data are indistinguishable - all the tools and techniques that work with data also apply to meta-data.

RDF is flexible and simple enough that it can describe queries, data models, user interfaces, validation rules, business processes, integration patterns and more.

Scorpio4 take those possibilities and provides the infrastructure and tooling needed to design, test, deliver, deploy and maintain sophisticated systems that rely upon smart data rather than smart code.
